# Agenda фасилитатора — 2.5 ч

Поминутный run-sheet для проведения воркшопа. Источник истины:
[charter.md](./charter.md). Детали рычагов: [hygiene-methods.md](./hygiene-methods.md),
[mcp-proxy-methods.md](./mcp-proxy-methods.md). Протокол участника:
[workshop-protocol.md](./workshop-protocol.md).

---

## Pre-flight (участники — до сессии)

```bash
git clone <repo>
npm install
npm run setup                 # Docker + MCP + variant-1 (одна команда)
npm run workshop:doctor       # Проверить что всё готово
```

**Codex-участники дополнительно:** `npm run setup:mcp` (регистрирует MCP-сервера глобально).

**Проверить:** `npm run workshop:doctor` — все 12 проверок должны быть ✓. Одна `!` (no TASK.md)
нормально — участник выберет вариант на месте.

**Фасилитатор до сессии:** прогоняет e2e-тесты, записывает референсные дельты.

## Timeline

### 0:00–0:15 — Интро + setup check (15 мин)

**Цель:** все могут запустить продукт, увидеть баги, запустить агент, прочитать замеры.

**Сделать:**
1. «Поднимите руку у кого Docker не запустился / npm start не работает» — помочь.
2. `npm start` → открыть http://localhost:4200 → показать баги в поиске.
3. `npx ccusage session` — показать что трекер читает локальные логи.
4. **Тезис воркшопа:** «Минимальная стоимость корректного результата. Не просто
   "меньше токенов", а дешевле при том же качестве. Quality gate решает.»
5. Объяснить 3-run структуру: baseline → гигиена → tool-layer.

**Слайд / доска:**
```
Run 1: Раздутый baseline  → замер
Run 2: Гигиена AGENTS.md  → замер → дельта
Run 3: Прокси / хуки       → замер → дельта
Wrap: Сравнение, потолок
```

**Три вектора багов (выбрать один):**

| Variant | Страница | Ticket | Тип задачи | Сложность |
|---|---|---|---|---|
| 1 | `/catalog` | JIRA-0321 | Data grid: URL sync, cache, SWR, cancellation | 🔴 Высокая |
| 2 | `/orders` | JIRA-0410 | Data grid: URL sync, debounce, cache, SWR | 🔴 Высокая |
| 3 | `/edit/:id` | JIRA-0455 | Forms: validation, 400-mapping, XSS, dirty state | 🟡 Средняя |

**Разбить на команды** (по 2–3 человека). Каждая команда получает свой вектор:
- **Variant 1 — Catalog** (`/catalog`): data grid, 5 багов, URL-driven архитектура. Самый сложный.
- **Variant 2 — Orders** (`/orders`): data grid, 5 багов, URL-driven + debounce. Аналог variant-1 на других данных.
- **Variant 3 — Edit-card** (`/edit/:id`): форма, 5 багов, validation + XSS + dirty state. Средняя сложность, смешанный gate (3 теста красные, 3 зелёные из коробки).

Распределить варианты равномерно. Минимум одна команда на каждый вектор — так
интереснее на wrap-е сравнить разные типы задач.

### 0:15–0:55 — Run 1: Раздутый Baseline (40 мин)

**Цель:** замерить стоимость решения задачи с раздутым AGENTS.md.

**Сделать:**
1. «Выберите вариант если ещё не сделали: `npm run variant -- <1|2|3>`»
2. «Запустите: `npm run workshop:run1`»
3. Скрипт сбросит сценарий, сделает снапшот ДО.
4. «Теперь откройте ваш агент из директории `apps/angular-demo/`. Напишите промпт —
   опишите задачу своими словами. Агент прочитает TASK.md, сходит в JIRA, найдёт баг,
   починит, проверит. Когда закончит — закройте агента и нажмите Enter.»
5. Участники работают. Фасилитатор ходит, помогает с промптами.
6. Когда все нажали Enter — скрипт показывает дельту. «Запишите свои цифры.»

**Что сказать во время Run 1:**
- «Не оптимизируйте AGENTS.md. Не меняйте модель. Это baseline.»
- «Пишите промпт как обычно.»
- «Важно: закройте агента перед тем как нажать Enter.»
- **Variant 1/2 (data grid):** «Агент должен прочитать TASK.md, потом сходить в JIRA за
  JIRA-0321/0410, потом в Confluence за контрактом. Это три шага в одном промпте.»
- **Variant 3 (edit-card):** «Агент должен найти XSS-уязвимость, добавить валидацию
  и 400-mapping. Подсказка: функция `validate()` уже есть в `editcard.types.ts`.»

**Ожидаемое время:** 25–35 мин на прогон агента, 5 мин на обсуждение цифр.

### 1:35–1:55 — Теория прокси / хуков (20 мин)

**Цель:** объяснить два подхода к оптимизации уровня инструментов.

**Показать документ:** `docs/mcp-proxy-methods.md` — 8 методов трансформации.

**Объяснить Option A — Proxy-MCP (10 мин):**
- Отдельный сервис между агентом и MCP-серверами
- Truncation, field filtering, structured summary
- Плюс: переиспользуется между агентами, запоминает состояние (dedup)
- Минус: дополнительный процесс

**Объяснить Option B — Local hooks (5 мин):**
- `.claude/hooks/` скрипт на PostToolUse
- Те же трансформации без отдельного сервиса
- Плюс: ноль инфраструктуры
- Минус: stateless, только Claude Code

**Сравнение (слайд):**

| | Proxy-MCP | Local hooks |
|---|---|---|
| Где | Отдельный сервис | `.claude/hooks/` |
| Инфра | +1 процесс | 0 |
| Состояние | Да (dedup) | Нет |
| Агенты | Любые | Claude Code |

**Сказать:** «Выберите подход и реализуйте его за следующие 20 минут. Scaffold есть —

## Риски и mitigation

| Риск | Что делать |
|---|---|
| Агент не решает задачу за 40 мин | Предложить более конкретный промпт. Если совсем не идёт — дать готовый промпт. |
| Gate FAIL на Run 2 или 3 | Оптимизация сломала решение. Откатить, применить меньше рычагов. |
| Codex/Cursor не измеряет токены | Для Codex — проверить `log_dir`. Для Cursor — gate-only, это норм. |
| Docker не работает у участника | MCP-моки не обязательны для Run 1–2. Для Run 3 — работает на машине фасилитатора. |
| Не хватает времени | Run 3 можно сократить: дать готовый прокси, участник только измеряет. |
| Участники путают ветки / снапшоты | `workshop:run*` всё делает сам. Участник только пишет промпт и жмёт Enter. |
